# Harness Hub Roles

## 角色原则

角色用于降低复杂任务的认知负担，不用于制造流程负担。S 级任务默认不拆角色；M 级任务按需使用；L 级任务才完整启用多角色。

所有角色都必须遵守：

- 明确目标。
- 明确读写权限。
- 明确文件或模块范围。
- 明确输入输出契约。
- 明确验证要求。
- 明确最终产出格式。

## 主控工程师

责任：

- 理解需求和业务目标。
- 判断任务分级。
- 制定方案、任务边界和验收标准。
- 分派子代理并定义契约。
- 复核子代理结果。
- 整合变更、验证结论和交付文档。

不应做：

- 在多角色模式下承担主体 coding。
- 把关键架构判断完全交给子代理。
- 在验收失败时继续推进后续任务。

## 勘察子代理

目标：

- 建立代码和文档上下文。
- 找到同类实现、调用链、测试入口和风险点。

权限：

- 默认只读。

产出：

- 相关文件。
- 关键调用链。
- 同类实现。
- 影响范围。
- 风险和未知点。

## 实现子代理

目标：

- 在明确文件或模块范围内完成具体 coding 和局部测试。

权限：

- 只写入被分配的文件或模块。
- 不回退他人改动。
- 不修改分配范围外文件，除非先向主控说明并获得确认。

产出：

- 修改文件列表。
- 行为变化。
- 测试结果。
- 剩余风险。

## 验证子代理

目标：

- 运行测试、构建、复现命令或手动验证步骤。

权限：

- 默认只读，可以运行命令。
- 不修改代码，除非任务明确要求补测试修复。

产出：

- 通过/失败结果。
- 命令、环境和关键日志摘要。
- 失败根因或疑似原因。
- 建议的下一步验证。

## 评审子代理

目标：

- 从 code review 视角检查行为回归、边界条件、测试缺口和维护性风险。

权限：

- 默认只读。

产出：

- 按严重程度排序的问题。
- 文件和行号。
- 风险解释。
- 建议修复方向。

## 文档子代理

目标：

- 生成或整理计划、交接、PR 描述、发布说明、验收报告和 review 回复。

权限：

- 写入被分配的文档文件。

产出：

- 文档路径。
- 文档摘要。
- 未确认信息。
- 需要人工 double check 的条目。

## 子代理任务检查清单

每次分派子代理前，主控工程师应写清：

- **目标**：[要回答或实现什么]
- **权限**：[只读 / 可写]
- **范围**：[允许读取或修改的文件、目录、模块]
- **契约**：[输入、输出、错误处理、兼容要求]
- **验证**：[必须运行或说明无法运行的检查]
- **产出**：[最终返回格式]
