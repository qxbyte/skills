# Harness Hub Architecture

## 定位

Harness Hub 是一套通用、简洁、可复用的工程交付操作层。它不绑定具体技术栈、业务领域或 AI 工具，而是为软件开发任务提供稳定的工作方式：理解需求、建立上下文、设计方案、实施变更、验证结果、沉淀验收文档。

`harness-hub/SKILL.md` 是 Codex 入口；`docs/harness/` 是长期工程规范；`harness-hub/references/` 是可复制的输出模板；`scripts/` 提供轻量自动化辅助。

## 设计原则

- **通用优先**：规则面向开发行业常见交付场景，不把某个项目的特殊习惯写成默认要求。
- **简洁优先**：先用少量文档、模板和脚本支撑稳定流程，避免过度平台化。
- **可复用优先**：计划、交接、验收、角色契约和验证矩阵应能迁移到不同仓库。
- **验收优先**：每个有效交付都应能回答“做了什么、如何证明、还有什么风险”。
- **项目记忆跟随项目**：复杂任务的计划和交接记录默认写入当前项目目录，让上下文不依赖单次对话。

## 系统边界

Harness Hub 负责：

- 把需求转成可执行工程任务。
- 选择合适的任务分级和协作模式。
- 定义角色分工、写入范围、行为契约和验收标准。
- 选择验证矩阵并记录验证结果。
- 生成计划、交接、验收、PR 描述、发布说明等交付文档。

Harness Hub 不负责：

- 替代业务负责人做范围决策。
- 替代 CI/CD 平台、测试框架或项目构建系统。
- 强行要求所有项目使用同一目录结构或同一技术栈。
- 为简单任务引入不必要的多代理编排。

## 工作对象

- **任务**：一个可交付、可验证的开发目标。
- **计划文件**：`plans/<task-slug>.md`，记录背景、目标、非目标、影响范围、任务拆解和验证计划。
- **交接文件**：`handoffs/<task-slug>.md`，记录完成状态、关键文件、决策、验证结果、剩余风险和继续工作的入口。
- **验收文档**：可以是计划文件中的验收部分、交接文件、PR 描述、发布说明或独立验收报告。复杂任务必须有明确验收记录。

## 生命周期

1. **接收需求**：识别目标、约束、输入输出、验收标准和风险。
2. **任务分级**：判断任务为 S、M 或 L，决定是否启用多角色协作。
3. **建立上下文**：阅读项目结构、同类实现、测试入口、构建配置和已有文档。
4. **制定计划**：复杂任务生成 `plans/<task-slug>.md`，轻量任务可在对话中给出短计划。
5. **实施变更**：按现有项目风格小步修改，避免侵入式重构。
6. **验证验收**：根据验证矩阵运行检查，记录证据、缺口和剩余风险。
7. **交接沉淀**：复杂任务生成 `handoffs/<task-slug>.md`，交付总结面向下一位工程师可继续工作。

## 与 AI 工具的关系

Harness Hub 的核心规范应独立于具体 AI 工具。Codex、Claude Code 或其他工具可以有各自 adapter，但共享同一套工程交付约定：

- 任务分级一致。
- 角色契约一致。
- 验证矩阵一致。
- 计划和交接文档一致。
