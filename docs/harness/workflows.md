# Harness Hub Workflows

## 总览

Harness Hub 工作流遵循一条主线：需求进入、上下文建立、方案设计、实现验证、验收交接。不同任务只是在这条主线上增减深度。

## 通用工作流

1. **需求接收**
   - 明确目标、用户价值、输入输出、约束和验收标准。
   - 标出高风险不确定点；能合理假设的继续推进，不能假设的向用户确认。

2. **任务分级**
   - 按 `quality-gates.md` 判断 S、M、L。
   - 选择直接执行、有限子代理协作或完整多角色模式。

3. **上下文勘察**
   - 阅读项目结构、README、构建配置、测试入口和同类实现。
   - 识别用户已有未提交改动，不覆盖不相关修改。

4. **计划与契约**
   - S 级任务可用短计划。
   - M/L 级任务建议生成 `plans/<task-slug>.md`。
   - 多角色任务必须写明角色、范围、输入输出、验证和产出。

5. **实现**
   - 使用项目既有模式。
   - 控制改动范围，避免无关重构。
   - 复杂任务按任务点串行推进，每个任务点先验收再进入下一步。

6. **验证**
   - 根据任务类型选择验证矩阵。
   - 记录已执行检查、失败检查、替代检查和剩余风险。

7. **验收与交接**
   - 输出面向用户的完成说明。
   - M/L 级任务建议更新 `handoffs/<task-slug>.md`。
   - 发布、PR 或交接场景应包含验收文档或验收报告。

## 需求设计 Spec 到 AI Coding 工作流

适用于“下班前做需求设计，交给 AI coding，第二天人工 double check”的节奏。

1. **人工设计 Spec**
   - 写清背景、目标、非目标、用户路径、接口/数据约束、验收标准和风险。
   - Spec 不追求冗长，但必须能让另一位工程师独立理解。

2. **AI 生成计划**
   - AI 读取 Spec 和项目上下文。
   - 生成 `plans/<task-slug>.md`，拆分任务点、写明验证矩阵和验收口径。

3. **AI Coding 与自动检查**
   - AI 按计划实现。
   - 根据验证矩阵运行测试、构建、文档检查或浏览器验证。
   - 所有验证结果写入交接文件。

4. **AI 生成交接与验收草稿**
   - 生成 `handoffs/<task-slug>.md`。
   - 生成 PR 描述、发布说明或验收报告草稿。

5. **第二天人工 Double Check**
   - 人工重点检查需求理解、范围控制、验收证据、剩余风险和用户体验。
   - 人工确认后再合并、发布或安排后续任务。

## 需求文档逐步开发工作流

当输入是 PRD、规格说明、长 issue 或多需求文档时：

1. 解析需求点、优先级、依赖、风险和验收标准。
2. 拆成可验收任务点。
3. 每个任务点执行：上下文、方案、实现、验证、验收、文档落地。
4. 当前任务点验收不通过时，禁止继续后续任务点。
5. 最后汇总整体交付、风险和未覆盖项。

## PR Review 工作流

1. 收集 review comments 和相关 diff。
2. 分类：必须修复、建议优化、需要讨论、非本轮范围。
3. 对必须修复项逐条实现和验证。
4. 生成 review 回复草稿，说明改动和验证证据。

## CI 失败工作流

1. 获取失败 job、日志和触发 commit。
2. 最小化复现失败。
3. 定位根因并做最小修复。
4. 运行对应验证；无法运行时记录原因和替代检查。
5. 交付 CI 修复说明和剩余风险。

## 发布与验收工作流

1. 对照验收标准检查功能、测试、文档和发布影响。
2. 生成验收文档，包含证据、未覆盖项、回滚方式和发布注意。
3. 人工或负责人确认后进入发布或合并。
