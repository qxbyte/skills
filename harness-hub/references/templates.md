## 技术方案模板

```markdown
**背景**
[当前问题、业务目标、触发场景]

**目标**
- [目标 1]
- [目标 2]

**非目标**
- [本次明确不做的事情]

**现状**
- [相关模块、接口、数据流、限制]

**方案**
- [核心设计]
- [关键接口或数据结构]
- [兼容性处理]
- [错误处理和可观测性]

**备选方案**
- 方案 A：[优点 / 缺点 / 放弃原因]
- 方案 B：[优点 / 缺点 / 放弃原因]

**影响范围**
- 代码：[模块/文件]
- 数据：[表/字段/迁移]
- 接口：[API/事件/契约]
- 运维：[配置/CI/CD/发布]

**测试计划**
- [单元测试]
- [集成测试]
- [手动验证]
- [回归范围]

**风险与回滚**
- 风险：[说明]
- 缓解：[说明]
- 回滚：[说明]
```

## 需求文档解析模板

```markdown
**文档来源**
[路径 / issue / PRD / 用户粘贴内容]

**需求点**
- [REQ-1] [需求说明]：优先级 [P0/P1/P2]，依赖 [无/REQ-x]
- [REQ-2] [需求说明]：优先级 [P0/P1/P2]，依赖 [REQ-1]

**验收标准**
- [REQ-1] [可验证结果]
- [REQ-2] [可验证结果]

**风险与缺口**
- [风险/不明确点]：[影响]，[处理方式]

**拆分原则**
- 先做：[原因]
- 后做：[原因]
- 暂不做：[原因]
```

## 任务拆解模板

```markdown
1. 建立上下文
   - 阅读 [模块/文件]
   - 确认 [接口/数据/测试]

2. 实现核心逻辑
   - 修改 [文件]
   - 保持 [兼容约束]

3. 接入调用路径
   - 修改 [入口/适配层]
   - 处理 [错误/权限/状态]

4. 补测试
   - 新增 [测试]
   - 覆盖 [边界/回归]

5. 验证与交付
   - 运行 [命令]
   - 输出 [PR 描述/发布说明]
```

## 逐步开发任务卡模板

```markdown
**任务点**
[T1：名称]

**来源需求**
[REQ-x]

**前置依赖**
[无 / T0 / 外部确认]

**本轮边界**
- 做：[明确实现范围]
- 不做：[排除项]

**工作流**
- 需求澄清：[结论/待确认]
- 上下文勘察：[文件/模块/同类实现]
- 方案：[关键设计]
- 实现：[写入范围和行为契约]
- 测试验证：[命令/手动步骤]
- 验收：[通过标准、验收负责人、证据]
- 文档落地：[PR 描述/任务记录/发布说明/交接文档]

**验收结论**
[通过 / 不通过]

**失败回环**
如果不通过，回到：[方案修正 / 实现修正 / 测试补齐]，禁止推进下一任务点。
```

## 子代理分工模板

```markdown
**主控工程师**
- 责任：[需求判断 / 方案 / 任务分派 / 接口契约 / 结果复核 / 整合 / 最终交付]
- 当前关键路径：[下一步必须由主控处理的事情]

**子代理 A：代码勘察**
- 目标：[要回答的问题]
- 只读范围：[目录/模块]
- 产出：[调用链 / 影响范围 / 同类实现 / 风险点]

**子代理 B：验证分析**
- 目标：[测试、CI 或复现问题]
- 命令：[可运行命令]
- 产出：[通过/失败结果、失败原因、建议修复点]

**子代理 C：实现**
- 目标：[具体 coding 任务]
- 写入范围：[明确文件/模块，避免与其他子代理冲突]
- 契约：[输入输出 / API / 状态 / 错误处理 / 兼容要求]
- 验证：[必须新增或运行的测试]
- 产出：[变更说明、测试结果、剩余风险]

**子代理 D：代码评审**
- 目标：[检查行为回归、边界、测试缺口]
- 范围：[文件/模块]
- 产出：[按严重程度排序的问题]

**整合标准**
- 主控复核所有子代理结论。
- 主控检查实现子代理是否遵守文件归属和接口契约。
- 冲突结论以代码事实和测试结果为准。
- 最终交付只包含已确认的变更、验证和风险。
```

## 子代理任务检查清单模板

```markdown
**子代理任务**

- 目标：[要回答或实现什么]
- 权限：[只读 / 可写]
- 范围：[允许读取或修改的文件、目录、模块]
- 契约：[输入、输出、错误处理、兼容要求]
- 验证：[必须运行或说明无法运行的检查]
- 产出：[最终返回格式]
- 冲突处理：[遇到范围外改动或不一致结论时如何回报主控]
```

## 验收文档模板

```markdown

## 需求来源

[Spec / issue / PRD / 用户请求 / PR review]

## 完成范围

- [已完成事项]

## 非范围

- [本轮未做事项]

## 验收标准

- [标准 1]
- [标准 2]

## 验证结果

| 检查项 | 命令或步骤 | 结果 | 证据 |
| --- | --- | --- | --- |
| [检查项] | `[命令]` | [通过/失败/未运行] | [摘要] |

## 未覆盖项

- [未覆盖项]：[原因]

## 剩余风险

- [风险]：[缓解方式]

## 人工 double check

- [ ] 需求理解正确
- [ ] 范围没有漂移
- [ ] 验收标准逐条覆盖
- [ ] 验证结果可信
- [ ] 剩余风险可接受
```

## Review 回复模板

```markdown
已处理。

变更：
- [针对评论做了什么]
- [补充测试或验证]

验证：
- [命令/结果]
```

如果不同意 review 建议，使用：

```markdown
这里我建议暂时不这样改，原因是 [工程原因/兼容性/范围控制]。
我已补充 [替代方案/注释/测试] 来覆盖这个风险。
```
