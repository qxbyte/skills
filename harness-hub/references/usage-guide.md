## 这个 skill 是什么

`harness-hub` 是一个面向项目开发工程师的 CLI coding agent skill。它让 CLI coding agent 按照真实工程交付方式工作：理解需求、阅读代码、设计方案、实现修改、补充测试、运行验证、处理 CI/PR 反馈，并最终输出清楚的交付说明。

这里的 Harness Hub 指“把需求连接到可验证工程交付的统一入口”。它默认不是某个特定厂商工具的专属 skill；如果任务明确涉及 Harness.io、Harness CI/CD 或部署流水线，CLI coding agent 会把它作为 CI/CD 工程场景处理。

从系统视角看，Harness Hub 是一套通用、简洁、可复用的工程交付操作层。`SKILL.md` 负责触发和行为约定，`docs/harness/` 负责长期规范，`plans/` 和 `handoffs/` 负责让复杂任务的记忆跟随项目走。

## 适合什么场景

适合使用这个 skill 的任务包括：

- 把产品需求或 issue 落地成代码实现。
- 在现有项目中新增功能、修复 bug、做小范围重构。
- 为开发任务补测试、补验证命令、补交付说明。
- 排查构建失败、测试失败、CI 失败、PR review 反馈。
- 设计技术方案、任务拆解、开发计划、测试计划。
- 准备 PR 描述、commit message、release note、交接文档。
- 作为项目开发工程师接手一个陌生仓库并推进任务闭环。
- 读取需求说明文档、PRD 或较长 issue，并拆成可逐项验收的开发任务。

不适合单独使用的场景：

- 纯闲聊或只需要概念解释。
- 大范围组织流程咨询，但不涉及具体工程交付。
- 用户明确只要某个专门 skill，例如 PDF、图片生成、OpenAI 官方文档检索。

## 如何触发

可以直接在请求中写：

```text
使用 Harness Hub，帮我实现这个需求：...
```

也可以用中文工程交付类表达自然触发：

```text
作为项目开发工程师，帮我把这个 issue 落地。
```

```text
帮我修复这个 bug，并补测试和验证说明。
```

```text
帮我处理 PR review 里的问题，跑通测试后给我交付总结。
```

```text
帮我设计技术方案并拆成可执行开发任务。
```

如果希望显式使用子代理，可以写：

```text
使用 Harness Hub，并结合子代理并行处理这个需求：...
```

```text
使用 Harness Hub 多角色模式：主代理负责方案、分派、整合和交付，具体 coding 交给实现子代理。
```

如果希望 CLI coding agent 像任务卡一样拆分并逐步推进，可以写：

```text
使用 Harness Hub，多角色模式，开启可观测 Checklist，按“勘察-方案-实现-验证-交付”逐步完成：...
```

## 推荐使用方式

最有效的请求通常包含四类信息：

1. **目标**：要实现什么、修复什么、优化什么。
2. **上下文**：相关文件、issue、PR、报错、日志、截图、用户路径。
3. **约束**：不能改什么、兼容要求、性能要求、交付时间、技术栈限制。
4. **验收**：如何判断完成、需要哪些测试、期望输出什么文档。

示例：

```text
使用 Harness Hub。目标：新增导出 CSV 功能。
上下文：现有导出逻辑在 src/export，列表页在 src/pages/orders。
约束：不要引入新依赖；保持现有权限模型。
验收：新增单元测试，能通过 npm test，并给我 PR 描述。
```

如果你只给一个简短需求，CLI coding agent 会先读仓库并做合理假设；遇到高风险不确定点时才会停下来问你。

如果你提供需求说明文档，建议同时说明：

1. **文档位置或正文**：文件路径、issue 内容、PRD 文本或链接。
2. **交付边界**：本轮必须完成哪些部分，哪些可以暂缓。
3. **验收口径**：测试命令、人工验收步骤、文档产物和发布要求。

## 显式可观测运行

Harness Hub 本质上是 CLI coding agent 的工程交付工作流，不是自动后台流水线。如果你希望明确知道它已经运行，并观察到任务拆分和执行进度，可以在提示词中加入这些要求：

```text
请显式声明你正在使用 Harness Hub。
请先输出任务计划 Checklist，并在执行过程中逐步更新状态。
每个阶段开始前说明当前阶段：需求判断、代码勘察、方案、实现、验证、交付。
如果启用子代理，请列出每个子代理的职责、文件范围、产出和验证要求。
最终交付时说明：改了什么、验证了什么、没验证什么、剩余风险。
```

要稳定触发 CLI coding agent 的任务拆分 Checklist，可以明确要求：

```text
请开启任务拆分 Checklist，像项目计划一样逐步推进，并在每完成一步后更新状态。
```

推荐完整指令：

```text
使用 Harness Hub 多角色模式处理这个任务。

请显式声明 Harness Hub 已启动。
请开启任务拆分 Checklist，并逐步更新状态。
主代理负责需求判断、方案设计、任务分派、结果复核和最终整合；
实现子代理负责具体 coding；
其他子代理按需负责代码勘察、测试验证和代码审查。

目标：...
上下文：...
约束：...
验收：...

交付时请说明：
1. 完成了什么
2. 修改了哪些文件
3. 运行了哪些验证
4. 未验证项和剩余风险
```

## 典型工作流

### 0. 任务分级

CLI coding agent 会先判断任务级别：

- S 级：小范围、低风险、验证简单，主代理直接完成。
- M 级：跨少量模块或需要测试/文档，可按需启用勘察、验证或评审子代理。
- L 级：高风险、跨多链路或多任务点，需要完整多角色模式，并生成计划和交接文档。

### 1. 需求接收

CLI coding agent 会先识别任务类型：

- 新功能：确认入口、用户路径、数据契约和验收标准。
- Bug 修复：复现路径、失败原因、最小修复面、回归测试。
- 重构：行为保持、影响范围、迁移方式、测试保护。
- CI/PR 修复：失败日志、触发条件、相关 commit、验证命令。
- 技术方案：背景、目标、备选方案、风险、里程碑。

### 2. 代码勘察

CLI coding agent 会优先查看：

- 文件结构和构建配置。
- README、测试命令、开发脚本。
- 与需求相关的模块、接口、模型、测试。
- 同类功能和项目既有风格。
- 当前 git 状态，避免覆盖用户已有修改。

### 3. 方案设计

方案会尽量短而可执行，通常包含：

- 要改哪些模块。
- 为什么这样改。
- 测试与验证方式。
- 风险和假设。

简单任务会直接推进实现；复杂任务会先给你更明确的步骤。

M/L 级任务建议生成 `plans/<task-slug>.md`，记录背景、目标、非目标、影响范围、任务拆解和验证计划。

### 4. 实现与验证

CLI coding agent 会按照现有项目习惯进行小范围修改，并根据风险选择验证层级：

- 小 bug：相关单测或目标测试文件。
- 共享逻辑：包级测试或更多回归测试。
- 构建配置：构建命令和 CI 等价命令。
- 前端交互：本地服务、浏览器检查、截图或交互验证。

### 5. 交付总结

完成后，CLI coding agent 会说明：

- 完成了什么。
- 修改了哪些关键文件。
- 跑了哪些测试或检查。
- 没跑成的验证和原因。
- 是否还有剩余风险或建议后续事项。

复杂任务建议生成 `handoffs/<task-slug>.md`，记录完成状态、关键文件、决策、验证结果和剩余风险。

## 验收文档

验收文档是 Harness Hub 的重要交付物。它回答三件事：做了什么、如何证明、还有什么风险。

S 级任务可以在最终回复中完成轻量验收说明；M 级任务建议写入计划或交接文件；L 级任务必须留下计划、交接和验收结论。

验收记录通常包含：

- 需求来源。
- 完成范围和非范围。
- 验收标准。
- 验证命令或人工步骤。
- 验证结果。
- 未覆盖项和剩余风险。
- 人工 double check 清单。

## 需求说明文档逐步开发

当输入是一份需求说明文档，Harness Hub 会先做需求解析，再逐项开发：

1. **解析文档**：识别需求点、依赖关系、优先级、验收标准、风险和缺失信息。
2. **拆成任务点**：把大需求拆成可交付、可验证、边界清楚的小任务。
3. **串行推进**：每个任务点都完成需求澄清、上下文勘察、方案、实现、测试验证、验收和文档落地。
4. **验收门禁**：只有当前任务点通过验收并完成必要文档后，才继续下一个任务点。
5. **失败回环**：如果验收不通过，先回到方案或实现修正，重新验证，不继续推进后续任务。

示例：

```text
使用 Harness Hub 多角色模式。请读取 docs/requirements/export.md，
先解析需求并拆分任务点，然后按任务点逐步开发。
每个任务点都要完成实现、测试、验收记录和文档落地；
只有验收通过后再继续下一个任务点。
```

## 子代理协作模式

Harness Hub 可以和 CLI coding agent 的子代理能力结合，适合任务比较大、信息面比较宽、验证成本比较高的工程工作。主代理相当于项目开发负责人或 tech lead，子代理相当于专项工程同伴。进入多角色模式后，具体 coding 默认交给实现子代理，主代理负责设计、分派、复核、集成和最终交付。

常见分工：

- **主控工程师**：理解需求、确定方案、定义模块边界和接口契约、分派实现任务、整合结果、给出最终交付。
- **代码勘察子代理**：搜索模块、梳理调用链、找同类实现、确认测试入口。
- **实现子代理**：在明确文件范围内完成具体 coding、测试补齐和局部验证。
- **验证子代理**：运行测试、分析失败日志、复现 bug、整理验证结论。
- **评审子代理**：检查边界条件、回归风险、缺失测试和行为不一致。
- **文档子代理**：生成 PR 描述、发布说明、交接文档和 review 回复草稿。

推荐在这些场景中启用：

- 陌生大型仓库，需要快速建立上下文。
- 一次改动横跨前端、后端、测试、CI 或文档。
- CI 失败日志很多，需要并行定位。
- PR review 评论较多，需要逐条处理。
- 高风险改动，需要独立评审视角。

不建议启用的场景：

- 单文件小改动。
- 需求还非常模糊，连目标都没有确定。
- 多个子代理必须同时编辑同一文件，容易产生冲突。
- 当前下一步完全依赖一个核心判断，需要主代理先完成方案判断。

每个子代理任务都应写清目标、权限、范围、契约、验证和产出，避免多个角色在同一文件或同一决策点上互相覆盖。

示例：

```text
使用 Harness Hub，并结合子代理处理。
主代理负责方案、任务分派、结果复核和最终整合；
子代理 A 勘察现有实现；
子代理 B 在指定文件范围内实现修复并补测试；
子代理 C 分析测试和 CI；
子代理 D 做代码审查。
目标：修复订单导出在空数据时崩溃的问题。
验收：补回归测试并给出 PR 描述。
```

## 协作约定

- CLI coding agent 会优先直接动手完成任务，而不是只给建议。
- 只有当你明确要求使用子代理、subagent、并行代理或多角色模式时，Harness Hub 才会启用子代理协作。
- 如果你要求可观测 Checklist，CLI coding agent 会先创建任务计划，并在关键步骤完成后更新状态。
- 对高风险操作，例如删除文件、重置分支、覆盖用户改动，会先请求确认。
- 如果仓库有未提交改动，CLI coding agent 会默认认为是你的工作，除非你明确要求清理。
- 如果命令因为权限、网络或环境限制失败，CLI coding agent 会说明失败原因，并在需要时请求授权。
- 如果你要求“review”，CLI coding agent 会进入代码审查视角，优先输出问题和风险。

## 输出物说明

根据任务需要，CLI coding agent 可以产出：

- 代码修改。
- 单元测试、集成测试、回归测试。
- 技术方案。
- 开发任务拆解。
- 测试计划。
- PR 描述。
- Review 回复建议。
- Release note。
- 交接文档。

详细模板见 `references/templates.md`。

## 最佳实践

- 把失败日志、截图、issue 链接或 PR 评论直接贴给 CLI coding agent。
- 明确哪些文件或模块不能改。
- 明确是否需要提交 commit、推送分支或创建 PR。
- 对大型需求，先让 CLI coding agent 输出任务拆解，再逐项实现。
- 对线上风险任务，要求 CLI coding agent 列出回滚策略和发布验证点。
