# Harness Hub

Harness Hub 是面向项目开发工程师的 CLI coding agent 工作流入口。它用于把需求推进到可验证的工程交付：理解需求、读取代码、设计方案、实现修改、补测试、运行验证，并输出清晰的交付总结。

## 如何触发

最直接的触发方式是在请求里显式写出 `Harness Hub`：

```text
使用 Harness Hub，帮我完成这个需求：...
```

也可以用工程交付类表达自然触发：

```text
作为项目开发工程师，帮我把这个 issue 落地。
```

```text
帮我修复这个 bug，并补测试和验证说明。
```

```text
帮我处理 PR review 里的问题，跑通测试后给我交付总结。
```

```text
帮我设计技术方案并拆成可执行开发任务。
```

也可以把 `/harness-hub` 作为稳定的命令前缀使用。它适合把自然语言需求、需求文档路径、上下文、约束和验收标准一起交给 CLI coding agent：

```text
/harness-hub 帮我修复订单导出在空数据时崩溃的问题，补回归测试并给出验证说明。
```

结构化写法：

```text
/harness-hub

目标：新增订单列表 CSV 导出功能。
上下文：现有导出逻辑在 src/export，列表页在 src/pages/orders。
约束：不要引入新依赖；保持现有权限模型。
验收：新增单元测试，通过 npm test，并给出 PR 描述。
```

如果需求已经写在文档里，可以直接给出路径：

```text
/harness-hub

需求文档：docs/requirements/export-orders.md
目标：按文档逐项实现并验收。
上下文：优先参考现有导出模块和订单列表页面。
约束：每个任务点验收通过后再继续下一个任务点。
交付：说明完成范围、修改文件、验证命令、未验证项和剩余风险。
```

如果希望启用子代理或多角色协作，需要显式写出：

```text
使用 Harness Hub 多角色模式：主代理负责方案、分派、复核和整合；实现子代理负责具体 coding。
```

## 如何确认工作流已经运行

如果你希望 Harness Hub 的运行过程显式、可观测，可以在提示词中加入：

```text
请显式声明你正在使用 Harness Hub。
请先输出任务计划 Checklist，并在执行过程中逐步更新状态。
每个阶段开始前说明当前阶段：需求判断、代码勘察、方案、实现、验证、交付。
如果启用子代理，请列出每个子代理的职责、文件范围、产出和验证要求。
最终交付时说明：改了什么、验证了什么、没验证什么、剩余风险。
```

CLI coding agent 的任务计划会显示类似“共 N 个任务，已经完成 M 个”的进度。要稳定触发这种逐步任务拆分，可以明确要求：

```text
请开启任务拆分 Checklist，像项目计划一样逐步推进，并在每完成一步后更新状态。
```

## 推荐完整指令

```text
使用 Harness Hub 多角色模式处理这个任务。

请显式声明 Harness Hub 已启动。
请开启任务拆分 Checklist，并逐步更新状态。
主代理负责需求判断、方案设计、任务分派、结果复核和最终整合；
实现子代理负责具体 coding；
其他子代理按需负责代码勘察、测试验证和代码审查。

目标：...
上下文：...
约束：...
验收：...

交付时请说明：
1. 完成了什么
2. 修改了哪些文件
3. 运行了哪些验证
4. 未验证项和剩余风险
```

一句话版：

```text
使用 Harness Hub，多角色模式，开启可观测 Checklist，按“勘察-方案-实现-验证-交付”逐步完成：...
```

## 多角色模式约定

进入多角色模式后，主代理作为 tech lead / orchestration 负责方案、分派、复核和整合；主体 coding 默认交给实现子代理。其他子代理可按需承担代码勘察、测试验证、CI 日志分析、代码审查或交付文档整理。

不建议为单文件小改动或非常简单任务强行启用多角色模式，除非你明确要求。

## 典型交付结果

完成后，Harness Hub 应该给出：

- 完成了什么。
- 修改了哪些关键文件。
- 运行了哪些测试或检查。
- 没有运行的验证和原因。
- 剩余风险或建议后续事项。

更完整的使用说明见 `references/usage-guide.md`。
